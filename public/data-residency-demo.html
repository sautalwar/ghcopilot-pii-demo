How to narrate this live

Start with the simple rule: Copilot only sees what enters the prompt envelope. Then ask the customer which envelope they care about — IDE chat, CLI, web chat, MCP tool output, or normal application runtime. Once the scenario is clear, the residency answer becomes concrete and defensible.

What to emphasize

The strongest enterprise message is layered governance: content exclusion, scoped environments, redaction in tools, local cleanup, and evidence-based validation. Instructions help, but enforcement lives outside natural language.

Security buyer translation

When a customer asks “Where does the data end up?” they usually mean one of three things: who sees it during processing, how long anything persists after the response, and what the developer must clean up locally. Make sure your answer covers all three.

Proof techniques to repeat

  • Developer Tools → Network for destination host evidence.
  • Output Panel for metadata-only logging proof.
  • state.vscdb and shell history for local persistence proof.
  • Runtime PII demo and canary record for database isolation proof.

What not to over-claim

  • Do not claim instructions are enforcement.
  • Do not claim MCP is safe by default without tool redaction.
  • Do not collapse local-storage risk into cloud-retention risk — they are different.
  • Do not confuse application runtime traffic with Copilot traffic.
Critical enterprise nuance: The MCP scenario is the exception that proves the rule. Standard Copilot chat does not go fetch runtime database rows by itself. But if your tool returns raw PII, the model sees raw PII because you explicitly placed it into the prompt context.

Tab 3 · The Truth About Instructions

Are .md instruction files reliable?

This is the honest answer tab. It acknowledges the customer's skepticism, explains why they are partially right, and replaces hand-wavy promises with a framework that prioritizes deterministic controls over probabilistic guidance.

Natural language guidance only
Honest answer: yes, the customer is partially right. Instruction files are not deterministic security controls. They bias the model. They do not turn an LLM into a strict policy engine.
Control Type Reliability Use for Why it matters

What improves compliance

  • Short, specific, imperative wording.
  • An explicit safe alternative instead of only a prohibition.
  • Few conflicting rules and less instruction sprawl.
  • Examples that reinforce the desired pattern.

What not to do

  • Do not rely on instructions alone for secrets, PII, or policy enforcement.
  • Do not bury critical rules inside enormous instruction files.
  • Do not confuse “often works” with “security boundary”.
  • Do not skip human review because the instruction file looked strict.

Live demo: test it yourself

Run the exact experiment customers ask for: add a logging rule, ask Copilot to write a function ten times, record the hit rate, then tighten the language by giving the model a specific replacement. The point is not to guess a universal compliance number. The point is to show the customer that better wording helps, but deterministic controls still matter more.
